# PRD — 성심당 스탬프 웹 (데모)

- 문서 버전: v1.1
- 작성일: 2026-07-07
- 범위: 화면 데모 (NFC 실제 연동·로그인·보상 지급은 본 버전 범위 외)
- 개발 환경: Cursor / 배포: Vercel / 백엔드·Firebase 미사용 (데이터는 브라우저 로컬 저장)

---

## 1. 개요

### 1.1 제품 요약
성심당 매장에서 베이커리·음료 구매 시 NFC 스탬프를 휴대폰에 태깅하면 스탬프가 1개씩 적립되는 모바일 웹 서비스. 이번 버전은 실제 NFC 연동 전 단계로, 전체 화면과 사용자 경험을 검증하기 위한 데모를 만든다.

### 1.2 배경
- 성심당은 대전 지역 기반의 강한 브랜드 충성 고객층을 보유
- 방문 적립 경험을 통해 재방문 동기와 지점 간 순환 방문(본점, DCC점, 롯데백화점점, 케익부띠끄 등)을 유도
- 기존 성심당 멤버십 앱에 이미 '스탬프' 메뉴가 존재하므로, 해당 앱의 무드와 자연스럽게 이어지는 경험이어야 함

### 1.3 목표
| 구분 | 내용 |
|---|---|
| 데모 목표 | 스탬프 적립 → 확인 → 히스토리 조회의 핵심 흐름을 화면으로 검증 |
| 사용자 가치 | 방문 기록이 쌓이는 재미, 지점별 방문 이력 확인 |
| 이후 확장 | NFC 태그(URL 방식) 연동, 로그인, 보상(쿠폰) 지급 |

### 1.4 이번 버전에서 하지 않는 것 (Out of Scope)
- 실제 NFC 태깅 연동 → 데모에서는 "스탬프 적립 시뮬레이션" 버튼 등으로 대체
- 로그인/회원가입 → 마이페이지는 닉네임 변경만 제공
- 보상(리워드) 지급 로직 → 보상 구조는 추후 결정. 단, 화면 구조는 보상이 추가되어도 수용 가능하게 설계
- 서버/데이터베이스 구축 → Firebase 등 백엔드 미사용. 모든 데이터는 브라우저 로컬 저장소(localStorage)에 보관

## 2. 사용자 정의

| 항목 | 내용 |
|---|---|
| 주 사용자 | 성심당 방문 고객 (대전 시민 + 관광객) |
| 사용 환경 | 매장 내 모바일 (세로 화면 기준, 375~430px) |
| 핵심 시나리오 | 계산 후 NFC 스탬프에 폰을 태깅 → 적립 팝업 확인 → (가끔) 히스토리에서 방문 기록 확인 |
| 대상 지점 | 성심당 전 지점 |

## 3. 화면 요구사항

### 3.1 메인 화면
**목적:** 내 스탬프 현황을 한눈에 보여주는 홈

| ID | 요구사항 | 우선순위 |
|---|---|---|
| M-1 | 현재 보유 스탬프 개수를 화면 상단에 크게 표시 | 필수 |
| M-2 | 스탬프 판(빵 모양 스탬프 아이콘 그리드)으로 적립 현황을 시각화. 채워진 칸/빈 칸 구분 | 필수 |
| M-3 | 최근 방문 지점명 + 방문 일시 표시 (예: "본점 · 7월 5일 방문") | 필수 |
| M-4 | 인사말 영역: "{닉네임}님, 안녕하세요" (기존 앱 톤 유지) | 필수 |
| M-5 | 하단 내비게이션: 홈 / 스탬프 내역 / 마이페이지 | 필수 |
| M-6 | (데모용) 스탬프 적립을 시뮬레이션하는 트리거 — 실제 서비스에서는 NFC 태깅으로 대체됨을 주석 처리 | 필수 |
| M-7 | 스탬프 판의 총 칸 수는 보상 구조 미정이므로 10칸 기본, 설정으로 변경 가능하게 | 권장 |

### 3.2 스탬프 적립 팝업
**목적:** 태깅 직후 적립 성공을 즉각적이고 기분 좋게 알림

| ID | 요구사항 | 우선순위 |
|---|---|---|
| P-1 | 적립 성공 시 전면 팝업(모달) 노출: "스탬프가 적립되었어요!" | 필수 |
| P-2 | 팝업에 적립 지점명 + 날짜 + 현재 누적 개수 표시 (예: "본점 · 5번째 스탬프") | 필수 |
| P-3 | 스탬프가 찍히는 애니메이션(도장 찍기 모션) 적용 — 프리미엄 완성도 기준 | 필수 |
| P-4 | 닫기 버튼 또는 배경 탭으로 닫힘 → 메인 화면의 스탬프 판에 반영된 상태로 복귀 | 필수 |
| P-5 | 동일 세션 중복 적립 방지 정책(예: 지점당 1일 1회)은 추후 정책 결정 — 데모에서는 제한 없음 | 참고 |

### 3.3 스탬프 내역 (달력)
**목적:** 월별 방문·적립 기록을 달력으로 확인

| ID | 요구사항 | 우선순위 |
|---|---|---|
| H-1 | 월 단위 달력 뷰. 좌우 화살표로 이전/다음 달 이동 | 필수 |
| H-2 | 적립이 있는 날짜에 스탬프 마크(빵 아이콘) 표시 | 필수 |
| H-3 | 날짜 탭 시 해당 일의 상세: 지점명, 적립 시각, 적립 개수 | 필수 |
| H-4 | 달력 하단에 해당 월 요약: "이번 달 방문 N회 · 스탬프 N개" | 권장 |
| H-5 | 적립 내역이 없는 달은 빈 상태(Empty State) 문구 표시 | 필수 |

### 3.4 마이페이지
**목적:** 최소한의 개인 설정 (로그인 기능은 추후)

| ID | 요구사항 | 우선순위 |
|---|---|---|
| Y-1 | 현재 닉네임 표시 + 변경 기능 (입력 → 저장) | 필수 |
| Y-2 | 닉네임 유효성: 1~10자, 공백만 입력 불가 | 필수 |
| Y-3 | 누적 스탬프 총 개수, 총 방문 횟수 요약 표시 | 권장 |
| Y-4 | 로그인/계정 영역은 "준비 중" 상태로 자리만 확보 | 참고 |

## 4. 디자인 가이드 (성심당 멤버십 앱 무드 준수)

기존 성심당 멤버십 앱 홈 화면을 기준으로 한다.

| 항목 | 가이드 |
|---|---|
| 배경색 | 크림/아이보리 계열 (기존 앱의 따뜻한 미색 배경) |
| 포인트 컬러 | 성심당 레드(로고 도장 컬러) + 브라운 계열 |
| 톤앤무드 | 1956년 헤리티지, 따뜻한 수채화 일러스트 감성, 복고적이면서 정갈함 |
| 타이포 | 세리프 계열 숫자 강조(예: "1956" 타이틀 느낌) + 본문은 가독성 좋은 고딕 |
| 아이콘 | 기존 앱의 라인 아이콘 스타일(포인트·스탬프·매장 아이콘 톤) 유지 |
| 스탬프 비주얼 | 빵 또는 성심당 도장 모티프. 찍힌 스탬프는 붉은 인주 도장 느낌 |
| 완성도 기준 | 임시 디자인·기본 템플릿 룩 금지. 여백, 정렬, 인터랙션 디테일까지 프리미엄 수준 |

## 5. 기술 스택 및 배포

| 항목 | 결정 사항 | 비고 |
|---|---|---|
| 개발 도구 | Cursor | AI 기반 개발, 비개발자 주도 |
| 프레임워크 | Next.js + React 권장 | Vercel 배포에 최적, 별도 서버 설정 불필요 |
| 스타일링 | Tailwind CSS 권장 | 디자인 가이드(4장) 커스텀 컬러로 세팅 |
| 백엔드/DB | 사용 안 함 (Firebase 미사용) | 서버 없이 프론트엔드만으로 동작 |
| 데이터 저장 | 브라우저 localStorage | 닉네임·스탬프 기록 저장. 아래 제약 참고 |
| 배포 | Vercel | GitHub 저장소 연결 시 자동 배포 |
| 대응 화면 | 모바일 웹 우선 (375~430px) | 데스크톱 접속 시에도 모바일 폭 중앙 정렬 |

**localStorage 제약 (알아둘 점)**
- 데이터가 사용자의 브라우저에만 저장됨 → 기기·브라우저를 바꾸면 기록이 보이지 않음
- 브라우저 데이터 삭제 시 스탬프 기록도 함께 삭제됨
- 데모 검증에는 충분하며, 정식 서비스 전환 시 백엔드 도입으로 해결 (7장 로드맵 참고)

## 7. 향후 로드맵 (참고)

1. **NFC 연동:** NFC 태그에 지점별 고유 URL을 심어, 태깅 → URL 접속 → 적립 처리 방식 적용
2. **백엔드 도입:** 정식 서비스 전환 시 서버·DB 구축 → 기록의 영구 보관 및 기기 간 동기화
3. **로그인:** 계정 연동 후 기기 간 스탬프 동기화
4. **보상 구조:** 적립 개수별 쿠폰/혜택 정책 확정 후 스탬프 판·팝업에 보상 진행률 반영
5. **부정 적립 방지:** 지점당 적립 횟수 제한, 위치 검증 등 정책 수립 (서버 필요 → 백엔드 도입 이후)
